> 记录时间：2026-07-06
> 目的：把已经落地、已经讨论、暂不做和后续规划集中记录，方便继续开发，也方便以后合并官方更新。

## 当前结论

我们目前围绕三条主线改造：

- 品牌与站点信息高度自定义，包括名称、Logo、favicon、首页、服务条款、隐私政策和多语言内容。
- 计费逻辑调整为“订阅额度优先，额度用完后继续按量扣余额”。
- 部署方式优先面向 Linux amd64，使用 GHCR 镜像和 Docker Compose，`deploy/` 目录作为本地部署资料，不提交到 Git。

## 已完成

### 品牌与首页

- 新增公开首页 `/`，避免用户第一次访问就被直接送到登录页。
- 登录页读取站点名称、描述和 Logo。
- 新增后台站点设置页面 `/admin/site`。
- 支持配置站点名称、短名称、描述、Logo、暗色 Logo、favicon、首页文案、页脚、联系方式、服务条款和隐私政策。
- 支持上传品牌素材，Logo 支持 `png`、`jpg`、`webp`、`svg`，favicon 支持 `png`、`jpg`、`webp`、`svg`、`ico`。
- 修复 title 在路由切换后又变回默认值的问题。
- 修复 favicon 初始加载和后续切换不一致的问题。
- 移除前台用户设置里的 `/setting/about` 入口，旧页面重定向。
- 侧边栏 Logo 点击默认跳转 `/chat`。
- 去掉浏览器控制台里的官方 ASCII banner。

### 法务与多语言

- 把原来的“用户协议”语义升级为“服务条款”和“隐私政策”。
- 注册、充值、订阅需要用户主动勾选同意。
- 登录可以默认勾选。
- 后端已增加协议同意校验，不能只靠前端绕过。
- 管理后台可分别配置服务条款和隐私政策。
- 支持多语言条款内容，包括英文、简体中文、繁体中文。
- 条款弹窗已改为 Markdown 渲染，标题、列表、段落层级会按文档样式展示。
- 新增繁体中文语言支持。

### 计费

- 周期套餐逻辑调整为：有订阅时优先消耗套餐周期额度，额度用尽后继续按量扣余额。
- 无订阅时按量扣余额，不再依赖旧的默认 `free` 套餐。
- 新安装不再默认创建旧语义的 `free` 套餐。
- 后台已支持新增订阅方案。
- 后台订阅方案创建时禁止使用保留编码 `free`。
- 修复套餐编码输入框在新版浏览器下的 `pattern` 正则兼容问题。
- 模型定价已从计费长页面里拆到独立 tab，后续模型多了会更好维护。
- 后台订阅方案支持上移、下移调整展示顺序，对应后端接口为 `POST /api/v1/admin/billing/plans/order`。
- 新增支付优惠码能力，支持余额充值优惠码和套餐购买优惠码。
- 优惠码支持适用范围、百分比折扣、固定金额抵扣、最低订单金额、最大优惠金额、限定套餐、总次数、单用户次数、过期时间和启停状态。
- 充值、在线购买套餐、余额购买套餐时都可以填写优惠码。
- 优惠码会写入订单快照，支付回调按订单快照发放权益，不重新计算折扣。
- 用户可以用账户余额购买订阅套餐，仍然需要勾选服务条款和隐私政策。
- 模型定价新增可选倍率，默认按 `1x` 处理；配置 `0.5` 表示按基础价格 50% 计费，配置 `2` 表示 2 倍计费。
- 模型倍率已参与真实账单计算，并写入用量快照，方便后续查账。

### 模型展示

- 模型新增可选展示名称。
- 展示名称为空时继续显示真实平台模型名。
- 后台模型列表、模型编辑、排序面板、前台模型选择器和 `@` 提及菜单已接入展示名称。
- 前台用户侧有展示名称时优先只显示展示名称，不再额外用小字展示真实请求模型名。

### AFF 一级返利

- 已新增一级邀请关系和返利记录表。
- 用户可在订阅页查看邀请码、邀请链接、返利比例、邀请人数和可提现返利金额。
- 邮箱注册会读取登录页 URL 中的 `ref`、`aff` 或 `invite` 参数并绑定一级邀请关系。
- 充值支付成功后按被邀请用户实际支付基础金额生成一级返利。
- 仅一级有效：A 再邀请 C 时，只有 A 获得返利，A 的邀请人不会继续获得返利。
- 邀请码无效时不阻断注册，只是不绑定邀请关系。

## 暂不做

### 新用户默认开通 trial 套餐

讨论过的方案：

- 新增套餐编码 `trial`，名称 `Free`，周期永久，额度 `0.3`。
- 用户注册后自动开通这个套餐，作为新用户体验额度。

当前决定：

- 先不加自动开通逻辑。
- `trial` 套餐可以保留为手动发放或后续再启用。
- 不建议继续使用 `free` 编码，因为系统里 `free` 已经有旧的特殊语义。

如果以后要做：

- 后台增加“新用户默认套餐”配置项。
- 注册链路和第三方登录首次注册链路都要覆盖。
- 后端创建用户时直接写入订阅记录，不能只靠前端处理。

## 部署判断

2 核 4G 环境可以先跑，但建议控制并发和 worker 数量。

推荐部署方式：

- 应用、PostgreSQL、Redis 都用 Docker Compose，维护成本最低。
- 如果追求数据库稳定和备份便利，可以 PostgreSQL 单独托管或本机安装，应用和 Redis 用 Docker。
- 目前我们的镜像只面向 Linux amd64，符合当前部署目标。

Docker 构建慢是正常的，所以日常部署优先拉 GHCR 镜像，不建议每次在服务器本地构建。
